# 优化区域争夺派兵动画 — 动态数字显示

## 问题

当前派兵流程中，士兵飞行时领地数字不变，直到士兵全部到达目的地后才一次性结算数字。这导致视觉上"士兵飞走了但数字没变"、"士兵到了数字突然跳变"的不自然体验。

**时间线分析：**
- 出发地：服务器立即扣兵（dispatchUnits line 380），但客户端等下一次 STATE 同步才看到（最多延迟 ~1s）
- 目的地：服务器 tickMovingTroops 批量结算（line 279/291），N个士兵同时+/-N

## 方案：客户端渐进式数字预测

**核心思路：** 客户端基于已知的士兵飞行时间线，在本地预测并逐步更新领地数字显示，不修改服务器权威逻辑。

### 涉及文件

| 文件 | 修改内容 |
|------|---------|
| `games/territory-control/src/services/gameView.js` | 新增 `getArrivalEvents()` 函数，计算每个士兵到达每个节点的时间 |
| `games/territory-control/src/components/GameScreen.vue` | 新增客户端数字预测系统：departure/arrival 事件驱动数字渐变 |

### 详细步骤

#### Step 1: gameView.js — 新增 `getArrivalEvents()` 函数

为每个 movingTroop 计算士兵到达每个节点（包括终点和中间节点）的时间事件。

```js
export function getArrivalEvents({
  movingTroops = [],
  animationStateById = {},
  now = Date.now()
}) {
  const events = []
  const streamDuration = 600
  const individualDuration = TRAVEL_TIME_PER_EDGE - streamDuration

  movingTroops.forEach(troop => {
    const stepStartTime = animationStateById[troop.id]?.stepStartTime ?? now
    const currentStep = troop.path[troop.currentStep]
    const nextStep = troop.path[Math.min(troop.currentStep + 1, troop.path.length - 1)]

    if (currentStep === nextStep) return

    for (let i = 0; i < troop.amount; i++) {
      const delay = troop.amount > 1 ? (i / (troop.amount - 1)) * streamDuration : 0
      const arrivalTime = stepStartTime + delay + individualDuration

      if (arrivalTime <= now) continue  // 已经到达

      // 士兵到达目标节点的事件
      events.push({
        troopId: troop.id,
        playerId: troop.playerId,
        targetTerritoryId: nextStep,
        sourceTerritoryId: currentStep,
        soldierIndex: i,
        arrivalTime,
        isFinalStep: troop.currentStep >= troop.path.length - 1,
        troopAmount: troop.amount
      })
    }
  })

  return events
}
```

#### Step 2: GameScreen.vue — 客户端数字预测系统

新增以下状态和逻辑：

**2a. 新增响应式状态**

```js
// 客户端预测的领地显示数字
const displayUnits = ref({})
// 已处理过的出发事件（避免重复扣减）
const processedDepartures = ref(new Set())
// 已处理过的到达事件（避免重复增加）
const processedArrivals = ref(new Set())
```

**2b. 出发扣减逻辑**

当 `movingTroops` 新增时，立即在客户端预测出发地数字减少：

```js
watch(movingTroops, (newTroops) => {
  newTroops.forEach(troop => {
    const key = troop.id
    if (!processedDepartures.value.has(key)) {
      processedDepartures.value.add(key)
      // 出发地数字已经在服务器端扣除
      // 但客户端可能还没收到同步，确保本地显示同步
      const sourceId = troop.path[0]
      if (displayUnits.value[sourceId] !== undefined) {
        displayUnits.value[sourceId] = Math.max(0,
          displayUnits.value[sourceId] - troop.amount)
      }
    }
  })
  // 清理已不存在的 troop 记录
  const validIds = new Set(newTroops.map(t => t.id))
  for (const id of processedDepartures.value) {
    if (!validIds.has(id)) processedDepartures.value.delete(id)
  }
  for (const id of processedArrivals.value) {
    if (!validIds.has(id)) processedArrivals.value.delete(id)
  }
}, { deep: true, immediate: true })
```

**2c. 到达增加逻辑（在 tickAnimation 中处理）**

```js
function tickAnimation() {
  const currentNow = Date.now()
  now.value = currentNow

  // 处理士兵到达事件
  const events = getArrivalEvents({
    movingTroops: movingTroops.value,
    animationStateById: localTroopAnim.value,
    now: currentNow
  })

  events.forEach(event => {
    const eventKey = `${event.troopId}-${event.soldierIndex}`
    if (processedArrivals.value.has(eventKey)) return
    processedArrivals.value.add(eventKey)

    const territoryId = event.targetTerritoryId
    if (displayUnits.value[territoryId] !== undefined) {
      // 增加 1 个士兵
      displayUnits.value[territoryId] += 1
    }
  })

  animFrame.value = requestAnimationFrame(tickAnimation)
}
```

**2d. 同步/校正逻辑**

当服务器 STATE 下发时，与客户端预测对比并校正：

```js
// 在 watch(territories) 中初始化 displayUnits
watch(territories, (newTerritories) => {
  newTerritories.forEach(t => {
    if (displayUnits.value[t.id] === undefined) {
      displayUnits.value[t.id] = t.units
    }
    // 如果当前没有正在飞向该领地的部队，直接同步服务器值
    const hasInboundTroops = movingTroops.value.some(
      troop => troop.path[troop.currentStep + 1] === t.id
    )
    if (!hasInboundTroops && !processedArrivals.value.size) {
      displayUnits.value[t.id] = t.units
    }
  })
}, { deep: true, immediate: true })
```

将 `{{ territory.units }}` 替换为 `{{ getDisplayUnits(territory) }}`：

#### Step 3: 出发地的即时视觉反馈

在士兵飞行动画中，出发地数字应该在士兵"离开"时就开始减少（而不是等到达目的地才变）。修改 `tickAnimation` 中的逻辑：当士兵的 `progress > 0.1`（已离开出生动画）时，出发地数字 -1。

```js
// 在 tickAnimation 中增加出发地扣减
const movingVisuals = getMovingTroopVisuals({
  movingTroops: movingTroops.value,
  territories: territories.value,
  animationStateById: localTroopAnim.value,
  now: currentNow
})

// 按 troop 分组，记录已"离开"出发地的士兵数量
const departedByTroop = new Map()
movingVisuals.forEach(v => {
  const troopId = v.id.split('-').slice(0, -1).join('-')  // 去掉士兵索引
  if (v.progress > 0.1) {
    departedByTroop.set(troopId, (departedByTroop.get(troopId) || 0) + 1)
  }
})

// 出发地扣减
movingTroops.value.forEach(troop => {
  const sourceId = troop.path[troop.currentStep]
  const departed = departedByTroop.get(troop.id) || 0
  if (departed > 0) {
    const originalUnits = territories.value.find(t => t.id === sourceId)?.units ?? 0
    const targetUnits = originalUnits - troop.amount
    const progress = Math.min(departed / troop.amount, 1)
    displayUnits.value[sourceId] = Math.round(
      originalUnits - troop.amount * progress
    )
  }
})
```

**注意：** 这段出发地扣减逻辑需要与到达增加逻辑协调，避免重复计算。更简洁的做法是：

### 简化方案（推荐）

不区分出发和到达，直接在 `tickAnimation` 中基于 `getMovingTroopVisuals` 的 `progress` 值计算每个领地的显示数字：

```
对于每个领地 L：
  displayUnits[L] = serverUnits[L]
                  - sum(该领地出发的部队中，已离开的士兵数)
                  + sum(该领地到达的部队中，已到达的士兵数)
```

这样避免了复杂的事件追踪，直接用视觉进度驱动数字。

### 边界情况

1. **服务器校正**：当服务器 STATE 下发时，如果某领地没有入站/出站部队，直接同步服务器值
2. **战斗结算**：中间格子发生战斗时，服务器批量结算。客户端只负责"到达"的视觉效果，战斗结果由服务器权威决定
3. **多个部队同时到达同一格子**：基于每个 troop 的 arrive events 独立计算
4. **MAX_UNITS 上限**：客户端显示数字也要 clamp 到 [0, 50]
5. **重连/恢复**：displayUnits 在每次服务器 STATE 下发时以服务器值为基准重建

## 验证方式

1. 启动游戏服务器 (`npm run server`)
2. 打开两个浏览器窗口，两个玩家加入同一房间
3. 玩家A派兵攻击玩家B的领地：
   - 观察出发地数字是否在士兵飞出时立即减少
   - 观察目标地数字是否在士兵逐个到达时逐步增加
   - 战斗结算后数字应与服务器权威值一致
4. 派兵到友方领地（中间路过），数字变化应同样渐进
5. 快速连续派兵，确认数字不会错乱
